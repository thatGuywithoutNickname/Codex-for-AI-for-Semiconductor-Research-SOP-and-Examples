# PINO Project SOP: Codex and ChatGPT Research Workflow

This document defines the standard operating procedures for using Codex and the web-based ChatGPT interface to plan, implement, verify, document, and iterate PINO research tasks.

The workflow is intentionally closed loop: a human and ChatGPT discuss the next task; ChatGPT generates a Codex-ready plan packet using the `PINO_GPT55_PLAN_SOURCE_GUIDE.md` format; Codex executes the packet; the human reviews Codex output; and the finalized report is uploaded back to ChatGPT as curated context for the next planning cycle.

## Scope

This SOP applies to all PINO repository work performed through a human/ChatGPT/Codex loop, including:

- code changes;
- documentation changes;
- experiment configuration and reporting;
- demo and assistant workflow changes;
- repository operations;
- decision records and current-reference updates.

This SOP does not replace the repository itself as the source of truth. Code, tests, configs, problem specifications, current references, and accepted reports remain authoritative over historical plans.

## Actors and Artifacts

| Term | Definition |
| --- | --- |
| Human reviewer | The person who owns research intent, resolves ambiguity, reviews Codex output, accepts or rejects results, and authorizes commits. |
| ChatGPT web | The planning and synthesis interface used for human discussion, plan packet generation, and next-cycle context building. |
| Codex | The execution agent that loads a standardized plan packet, edits the repository, runs verification, and produces a closeout report. |
| Plan packet | A Codex-ready task packet generated by ChatGPT from human discussion and the `PINO_GPT55_PLAN_SOURCE_GUIDE.md` format. |
| Finalized report | The curated closeout report accepted by the human after Codex execution. This is uploaded to ChatGPT as context for subsequent modifications. |
| Current truth | The latest accepted state of the repository and its durable references, not unreviewed Codex output or archived plans. |

## Closed-Loop Operating Model

```text
Human + ChatGPT task discussion
        ↓
ChatGPT generates standardized Codex plan packet
        ↓
Codex loads packet and current repository context
        ↓
Codex implements, verifies, and reports
        ↓
Human reviews output and finalizes report
        ↓
Human uploads finalized report to web ChatGPT
        ↓
ChatGPT uses the report as curated context for the next plan
```

The standardized plan format reduces variability at the Codex execution boundary. The finalized report upload reduces variability at the next ChatGPT planning boundary.

## SOP 0: Core Change Principles

**Purpose:**  
Ensure every change is scoped, reviewable, evidence-backed, and safe to commit.

**Applies to:**  
All code, documentation, experiment, configuration, app, assistant, and workflow changes.

**Principles:**

1. **Think before coding**
   - State assumptions explicitly.
   - Surface ambiguity and tradeoffs.
   - Ask when requirements are unclear.
   - Do not silently choose between materially different interpretations.

2. **Keep changes simple**
   - Implement the minimum solution that satisfies the accepted task.
   - Avoid speculative flexibility, unused abstractions, and broad refactors.
   - Do not add features that were not requested in the active plan.

3. **Make surgical edits**
   - Touch only the files required by the task.
   - Match existing repository style.
   - Clean up only unused code introduced by the current change.
   - Do not remove unrelated dead code unless explicitly requested.

4. **Define success criteria**
   - Every task must have a verification path.
   - Bug fixes should be reproduced by tests where practical.
   - Refactors must preserve behavior and pass relevant tests.
   - Experiment changes must define dry-run, smoke, or full-run acceptance gates.

5. **Separate execution from acceptance**
   - Codex output is not accepted merely because it was generated.
   - Human review is required before committing or using results as next-cycle truth.
   - The finalized report must distinguish accepted facts from unverified observations.

## SOP 1: Closed-Loop Planning and Handoff Workflow

**Purpose:**  
Define the full human/ChatGPT/Codex loop, including the missing step of uploading the finalized Codex report to web ChatGPT for subsequent modifications.

**Procedure:**

1. **Discuss the task with ChatGPT**
   - Define the research or implementation objective.
   - Identify baseline behavior, constraints, invariants, non-goals, and risks.
   - Identify required verification gates before implementation begins.
   - Resolve ambiguity before generating the plan packet.

2. **Generate a Codex-specific plan packet**
   - Use `PINO_GPT55_PLAN_SOURCE_GUIDE.md` as the plan-packet contract.
   - Generate a packet for Codex execution, not a general discussion memo.
   - Use the required packet directory pattern:

     ```text
     docs/plans/active/P-YYYYMMDD-short-slug/
     ```

   - Include the seven required files:
     - `PLAN.md`
     - `CONTEXT_LOAD.md`
     - `CODEX_PROMPT.md`
     - `ACCEPTANCE_GATES.md`
     - `DECISION_IMPACT.md`
     - `REFERENCE_UPDATES.md`
     - `CLOSEOUT.md`
   - Add optional appendices only when they materially reduce implementation ambiguity.

3. **Load the packet in Codex**
   - Start with `CONTEXT_LOAD.md`.
   - Then load `PLAN.md`, `CODEX_PROMPT.md`, and `ACCEPTANCE_GATES.md`.
   - Load appendices only when the task scope requires them.
   - Treat the packet as execution intent, not durable project truth.

4. **Execute and verify in Codex**
   - Make surgical repository changes.
   - Run the smallest sufficient verification gates first.
   - Broaden verification only when the changed surface justifies it.
   - Record every skipped gate with a reason.

5. **Produce the finalized report**
   - Summarize scope, changes, files touched, verification, evidence, decision impact, reference updates, and deferred items.
   - Mark whether the work is implemented, verified, blocked, partially complete, or rejected.
   - Identify any generated artifacts that remain local.
   - Identify whether a commit was created, previewed, or not attempted.

6. **Human review and acceptance**
   - Compare Codex output against the active plan and acceptance gates.
   - Accept, reject, or request revision.
   - Do not treat the report as next-cycle context until the human has reviewed and finalized it.

7. **Upload the finalized report to web ChatGPT**
   - Upload the human-reviewed finalized report to the web-based ChatGPT interface before requesting the next modification plan.
   - Upload supporting accepted documents only when needed, such as updated current references, decision records, evidence reports, or relevant diffs.
   - Do not upload raw logs, credentials, private data, large generated result folders, or unreviewed scratch output.
   - Tell ChatGPT to treat the report as curated context for the next planning cycle while still verifying durable truth against repository files.
   - State the next objective and ask ChatGPT to generate the next Codex plan packet using the standard packet format.

8. **Start the next cycle**
   - ChatGPT uses the uploaded finalized report plus current repository truth to generate the next plan packet.
   - Codex executes only after the next plan packet is accepted by the human.

**Handoff rule:**  
The finalized report is the bridge from Codex execution back to ChatGPT planning. It should be concise enough to load into ChatGPT, but complete enough to prevent the next plan from rediscovering already-resolved context.

## SOP 2: Codex Plan Packet Intake Workflow

**Purpose:**  
Ensure Codex starts from a complete, standardized, and task-specific plan packet.

**Procedure:**

1. **Confirm packet identity**
   - Verify the packet folder, frontmatter, and title all use the same packet ID.
   - Verify the packet ID follows `P-YYYYMMDD-short-slug`.
   - Verify the packet is under `docs/plans/active/` unless explicitly marked as backlog, blocked, superseded, or archived.

2. **Confirm required files exist**
   - `PLAN.md`
   - `CONTEXT_LOAD.md`
   - `CODEX_PROMPT.md`
   - `ACCEPTANCE_GATES.md`
   - `DECISION_IMPACT.md`
   - `REFERENCE_UPDATES.md`
   - `CLOSEOUT.md`

3. **Read in the required order**
   - `CONTEXT_LOAD.md`
   - `PLAN.md`
   - `CODEX_PROMPT.md`
   - `ACCEPTANCE_GATES.md`
   - Scope-specific appendices

4. **Check packet quality before editing**
   - Scope and non-goals are explicit.
   - Expected touched files are plausible and bounded.
   - Acceptance gates are runnable or manually checkable.
   - Decision impact uses the expected decision level and typed decision category.
   - Reference updates are explicit.
   - Stop conditions are clear.

5. **Stop if the packet is unsafe or underspecified**
   - Stop when required files are missing.
   - Stop when the packet conflicts with current repository truth.
   - Stop when acceptance gates cannot be interpreted.
   - Stop when the task requires scientific claims without sufficient planned evidence.
   - Ask the human for clarification or request a corrected ChatGPT packet.

**Rule:**  
Codex may refine execution steps inside the accepted packet scope. Codex must not silently change the packet’s objective, non-goals, evidence standard, or decision impact.

## SOP 3: Standard Change Workflow

**Purpose:**  
Define the default workflow for implementing any PINO project change.

**Procedure:**

1. **Load current context**
   - Start from current truth, not historical plans.
   - Prefer:
     - `docs/reference/`
     - `docs/context/packs/`
     - relevant code, tests, and configs
   - Treat archived plans as historical only.

2. **State assumptions and scope**
   - Identify what is in scope.
   - Identify what is explicitly out of scope.
   - Name affected subsystems: core model, experiments, demo, docs, assistant, audit, or workflow.

3. **Plan verification**
   - Choose the smallest verification set that matches the risk:
     - Python compile for syntax-only or script changes.
     - Focused pytest for behavior changes.
     - Dry-run for experiment runner or config changes.
     - Full-seed training/evaluation for accepted scientific results.

4. **Implement surgically**
   - Put importable implementation code in `pino/`.
   - Put app entrypoints in `apps/`.
   - Put runnable commands in `scripts/`.
   - Put experiment config in `configs/`.
   - Put project knowledge in `docs/`.

5. **Verify**
   - Run focused checks first.
   - Broaden only when the changed surface justifies it.
   - Record any skipped checks and why.

6. **Update documentation if project truth changed**
   - Update current references for durable behavior changes.
   - Update context packs if future agents need the new context.
   - Update reports for evidence.
   - Add decision records only for lasting tradeoffs.

7. **Do not commit until accepted**
   - Normal implementation does not automatically imply a project commit.
   - Commits require explicit human acceptance of the relevant results.

## SOP 4: Repository Layout Workflow

**Purpose:**  
Keep the project structure predictable and prevent source, generated artifacts, and project knowledge from mixing.

**Canonical layout:**

| Path | Purpose |
| --- | --- |
| `pino/` | Importable implementation code. |
| `apps/` | Application entrypoints, including Streamlit UI code. |
| `scripts/` | Runnable commands and workflow helpers. |
| `configs/` | Versioned experiment and suite configuration. |
| `docs/` | Project knowledge, references, reports, decisions, plans, and generated docs. |
| `tests/` | Regression, contract, smoke, architecture, and subsystem tests. |
| `results/` | Local generated experiment, audit, demo bundle, and validation outputs. Ignored by Git. |
| `figures/` | Local generated standalone figures. Ignored by Git. |

**Rules:**

- Do not place generated outputs under `scripts/`.
- `scripts/results/` and `scripts/figures/` are invalid legacy locations.
- Do not commit raw generated artifacts unless the human explicitly requests a specific artifact.
- Evidence from generated artifacts should be summarized in reports or decision records.

## SOP 5: Experiment Workflow

**Purpose:**  
Make experiment campaigns reproducible, reviewable, and separated from ad hoc script logic.

**Applies to:**  
Training runs, ablations, baselines, suite specs, roadmap experiments, and result reporting.

**Procedure:**

1. **Use suite specs for campaign intent**
   - Experiment suites live under `configs/suites/`.
   - Suite YAML should own variants, seeds, splits, output roots, and reporting context.
   - Scripts should delegate to suite runner logic instead of encoding campaign semantics directly.

2. **Use stable runners**
   - General suite runner:

     ```powershell
     python scripts/run_suite.py --suite <suite.yaml> --dry-run
     ```

   - Suite report runner:

     ```powershell
     python scripts/report_suite.py --suite <suite.yaml>
     ```

3. **Dry-run before training**
   - Dry-runs should expose planned run IDs, variants, seeds, splits, and output paths.
   - A dry-run is a workflow/code gate, not scientific evidence.

4. **Run full experiments when evidence is needed**
   - Scientific acceptance requires completed relevant training/evaluation runs.
   - Smoke runs and one-epoch runs do not establish promotion or rejection evidence.

5. **Store outputs locally**
   - Generated experiment outputs belong under root `results/`.
   - Do not commit raw result folders.

6. **Summarize accepted evidence**
   - Use `docs/reports/` for evidence summaries.
   - Use SDRs or legacy ADRs for scientific promotion/rejection decisions.
   - Update current references only when the accepted project truth changes.

**Common checks:**

```powershell
python -m compileall pino scripts
python scripts/run_suite.py --suite <suite.yaml> --dry-run
python scripts/report_suite.py --suite <suite.yaml>
pytest -q <focused tests>
```

## SOP 6: Documentation Workflow

**Purpose:**  
Keep project documentation organized by truth level and prevent historical plans from becoming accidental current truth.

**Documentation truth hierarchy:**

1. Executable truth: code, tests, configs, problem spec.
2. Current references: `docs/reference/`.
3. Context packs: `docs/context/packs/`.
4. Decision records: `docs/decisions/` and legacy `docs/adr/`.
5. Reports: `docs/reports/`.
6. Active plans: `docs/plans/active/`.
7. Finalized reports uploaded to ChatGPT for next-cycle context.
8. Archived plans: `docs/plans/archive/`.

**Document roles:**

| Path | Role |
| --- | --- |
| `docs/reference/` | Current truth summaries. |
| `docs/context/packs/` | Compact agent-loadable subsystem context. |
| `docs/decisions/` | Durable typed decision records. |
| `docs/adr/` | Legacy ADRs retained during migration. |
| `docs/reports/` | Evidence, reviews, comparisons, accepted result summaries, and finalized handoff reports. |
| `docs/plans/` | Implementation intent and working memory. |
| `docs/_generated/` | Generated indexes, manifests, stale-doc reports, and architecture atlas output. |

**Plan lifecycle:**

- Active plans live under `docs/plans/active/`.
- Backlog ideas live under `docs/plans/backlog/`.
- Blocked plans live under `docs/plans/blocked/`.
- Superseded plans live under `docs/plans/superseded/`.
- Completed historical plans live under `docs/plans/archive/YYYY/<packet>/`.

**Rules:**

- Do not place plan packets directly under `docs/plans/`.
- Do not treat archived plans as current truth.
- Do not treat an uploaded ChatGPT handoff report as committed repository truth unless the report states that the corresponding repository changes were accepted and committed.
- Do not duplicate legacy ADR bodies into typed decision folders during routine work.
- Regenerate generated docs when source docs change.

**Common docs checks:**

```powershell
python scripts/docs/generate_doc_manifest.py --output docs/_generated/doc_manifest.yaml
python scripts/docs/generate_decision_index.py --output docs/_generated/decision_index.md
python scripts/docs/build_agent_context.py --scope docs_workflow --output docs/context/active_context.md
python scripts/docs/generate_project_tree.py --check
python scripts/docs/lint_docs.py
python scripts/docs/generate_architecture_atlas.py --check
pytest -q tests/test_docs_tooling.py
pytest -q tests/test_architecture_atlas.py
```

Use a project-specific Python path only when the environment requires it.

## SOP 7: Decision Record Workflow

**Purpose:**  
Record durable project decisions without creating excessive documentation ceremony.

**Create a decision record when a change establishes a lasting tradeoff, such as:**

- model selection;
- scientific promotion or rejection;
- architecture/API boundaries;
- workflow policy;
- data contract;
- interface behavior;
- assistant guardrails;
- repository operations.

**Do not create a decision record for:**

- small bug fixes;
- routine cleanup;
- narrow implementation changes;
- evidence-only reports with no durable decision;
- purely local or temporary work.

**Typed decision categories:**

| Type | Use for |
| --- | --- |
| `SDR` | Scientific model, data, loss, optimizer, evaluation, or promotion decisions. |
| `ADR` | Software architecture, APIs, package boundaries, artifact contracts. |
| `IDR` | Interface and Streamlit workflow decisions. |
| `ASR` | Assistant behavior, tools, provider policy, guardrails. |
| `ODR` | Repository workflow, documentation, testing, and human-agent process. |

**Procedure:**

1. Start from `docs/decisions/decision_record_template.md`.
2. Choose the correct record type.
3. Fill metadata completely.
4. Describe context, drivers, considered options, chosen option, consequences, and evidence.
5. Link related reports, plans, tests, or generated evidence.
6. Regenerate decision indexes and run docs lint if needed.

**Important rule:**  
Scientific promotion or rejection requires an SDR or legacy ADR metadata update.

## SOP 8: Evidence and Reporting Workflow

**Purpose:**  
Preserve the reasoning and results behind accepted project states without committing raw generated artifacts.

**Procedure:**

1. **Run the required verification**
   - For code: tests, compile, and smoke checks.
   - For experiments: dry-runs plus full runs when needed.
   - For scientific acceptance: full-seed training/evaluation evidence.

2. **Summarize results**
   - Use `docs/reports/` for evidence reports and finalized handoff reports.
   - Include:
     - run scope;
     - completed and missing runs;
     - key metrics;
     - caveats;
     - acceptance or non-promotion statement;
     - verification commands;
     - skipped checks with reasons;
     - local generated artifact paths that were not committed.

3. **Update current truth only when accepted**
   - `docs/reference/current_model_summary.md` for model truth.
   - `docs/reference/current_interface_summary.md` for interface truth.
   - `docs/reference/project_structure.md` for layout, artifact, docs, or command truth.
   - `docs/reference/project_tree.md` when file layout changes.
   - Relevant context packs when future task routing requires the new context.

4. **Prepare the ChatGPT handoff**
   - Convert the Codex closeout into a concise finalized report.
   - Remove secrets, credentials, private paths, and unrelated logs.
   - Mark whether evidence is accepted, partial, rejected, or deferred.
   - Mark whether repository changes were committed, not committed, or only previewed.
   - Upload this finalized report to web ChatGPT before requesting the next plan.

5. **Do not commit raw outputs**
   - Keep `results/` and `figures/` local.
   - Commit summarized evidence instead.

## SOP 9: Accepted-Results Commit Workflow

**Purpose:**  
Create safe, repeatable project commits only after relevant results are accepted.

**Commit gate:**  
A project commit is allowed only after the human explicitly confirms that full-seed training/evaluation results, or other relevant final results, are accepted.

**Procedure:**

1. **Confirm acceptance**
   - Do not commit temporary checkpoints or unaccepted experiment states.
   - Commit one accepted task at a time.
   - Confirm the finalized report reflects the accepted state.

2. **Use the accepted commit helper in preview mode**

   ```powershell
   $PythonExe = "<path-to-python.exe>"

   powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\git\commit_accepted.ps1 `
     -Preview `
     -Message "<commit message>" `
     -Summary "<short summary>" `
     -Evidence "<accepted evidence summary>" `
     -DecisionPath "<optional decision record path>" `
     -PythonExe $PythonExe
   ```

   If Python is already on `PATH`, use the repository-supported command form and omit environment-specific path details where possible.

3. **Inspect preview**
   - The script updates `docs/reports/change_log.md`.
   - It clears the staging area.
   - It stages only allowlisted core paths.
   - It rejects generated/local-only paths.
   - It compiles staged Python files.
   - It prints staged files and diff stat.
   - In preview mode, no commit is created.

4. **Commit after preview passes**

   ```powershell
   $PythonExe = "<path-to-python.exe>"

   powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\git\commit_accepted.ps1 `
     -Message "<commit message>" `
     -Summary "<short summary>" `
     -Evidence "<accepted evidence summary>" `
     -DecisionPath "<optional decision record path>" `
     -PythonExe $PythonExe
   ```

5. **Update the finalized report**
   - Add the commit hash if a commit was created.
   - If no commit was created, state `Commit status: not committed`.
   - Upload the updated finalized report to web ChatGPT before the next planning cycle.

**Allowed staged areas include:**

- `.gitignore`
- `AGENTS.md`
- `pino/`
- `scripts/`
- `docs/`
- `configs/`
- `tests/`
- `apps/streamlit_demo/`

**Blocked paths include:**

- `data/`
- `results/`
- `figures/`
- `scripts/results/`
- `scripts/figures/`
- `.idea/`
- caches
- logs
- Python bytecode

**Outputs:**

- Updated `docs/reports/change_log.md`.
- Validated staged core files.
- Optional linked decision record.
- One accepted Git commit, when authorized.
- Updated finalized report for ChatGPT handoff.

## SOP 10: Closeout Workflow

**Purpose:**  
Ensure completed work leaves the repository in a usable, documented, verifiable, and reusable state for the next planning cycle.

**Procedure:**

1. Confirm implementation matches the requested scope.
2. Run relevant verification gates.
3. Summarize evidence in reports if results matter.
4. Update current references/context packs if durable truth changed.
5. Add a decision record only if the change creates a lasting tradeoff.
6. Archive completed active plans when appropriate.
7. Leave generated artifacts local unless explicitly requested.
8. Use the accepted-results commit workflow only after human acceptance.
9. Prepare the finalized report using [Appendix A](#appendix-a-finalized-report-template-for-chatgpt-handoff).
10. Upload the finalized report to web ChatGPT as next-cycle context before requesting another plan.

**Closeout checklist:**

- Scope satisfied.
- Tests/checks run or skipped with reason.
- No unrelated files changed.
- No raw generated artifacts staged.
- Evidence summarized.
- Decision record added only if warranted.
- Current docs updated if project truth changed.
- Commit made only after explicit acceptance.
- Finalized report reviewed by the human.
- Finalized report uploaded to ChatGPT for the next cycle.

- Use the canonical filename, such as `docs/workflows/codex_chatgpt_sop.md` or `docs/reference/codex_chatgpt_sop.md`.
- Link to `PINO_GPT55_PLAN_SOURCE_GUIDE.md` from the repository location where both files will live.
- Remove local absolute paths unless they are examples with placeholders.
- Remove secrets, credentials, private dataset paths, and machine-specific tokens.
- Confirm Markdown headings are unique and linkable.
- Confirm fenced code blocks specify a language where useful.
- Confirm commands match the repository’s current scripts.
- Run markdown linting if available.
- Review the rendered GitHub page for table and anchor formatting.
